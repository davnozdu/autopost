# Инструкция по эксплуатации autopost

Сервис собирает новости из RSS, отдаёт их LLM (Hermes или DeepSeek) для
написания статьи и показывает результат на согласование. Полный пайплайн:

**Источники (RSS) → «Собрать» → «Обработать» (LLM) → «Превью» → «Одобрить».**

---

## 1. Запуск

### Вариант A — Docker (рекомендуется)

```bash
# 1. Подготовьте переменные окружения
cp .env.example .env
#    впишите минимум: LLM_PROVIDER, LLM_KEY, SECRET_KEY
#    для Hermes дополнительно LLM_BASE_URL

# 2. Запустите
docker compose up -d --build      # или из готового образа GHCR (см. README)

# 3. Откройте админку
#    http://localhost:8080
```

`docker-compose.yml` уже монтирует нужные каталоги (см. §3).

### Вариант B — локально (без Docker)

```bash
python3.11 -m venv .venv && . .venv/bin/activate
pip install -r requirements.txt
cp .env.example .env               # заполнить
uvicorn app.main:app --port 8080
```

## 2. Первый вход и пароль

Админка **защищена паролем**. Пароль **задаётся при первом входе**, не в `.env`.

1. При первом открытии любой страницы вас перенаправит на **`/setup`**.
2. Введите и повторите пароль (минимум 6 символов) → «Сохранить и войти».
3. Дальше вход через **`/login`**, выход — кнопка «Выйти» в шапке.

Технически пароль хранится в БД как PBKDF2-хэш с солью (не в открытом виде).
Сессия — в подписанной cookie (ключ — `SECRET_KEY`).

> **Забыли пароль?** Сбросьте его, очистив поля в БД (см. §4.1) — после этого
> при следующем входе снова откроется `/setup`.

> **`SECRET_KEY`** должен быть стабильным. Если его поменять — все активные
> сессии разлогинятся (пароль при этом не теряется).

## 3. Какие директории подключать (volumes)

| Хост | Внутри контейнера | Назначение | Обязательно |
|------|-------------------|------------|-------------|
| `./data` | `/app/data` | БД + собранные новости (всё состояние) | **Да** — иначе данные пропадут при перезапуске |
| `./templates_sites` | `/app/templates_sites` | Шаблоны файлов сайта для будущей публикации | Желательно |

В `docker-compose.yml` это уже прописано:

```yaml
volumes:
  - ./data:/app/data
  - ./templates_sites:/app/templates_sites
```

Каталог данных можно переназначить переменной `DATA_DIR` (по умолчанию `data`).

## 4. Где что хранится

### 4.1. Конфигурация и пароль — `data/autopost.db`

Это файл **SQLite**. В нём лежит **всё состояние приложения**:

- **источники RSS** (таблица `feed`);
- **настройки обработки** LLM — язык, число символов, политика картинок,
  инструкции (таблица `appconfig`, одна строка `id=1`);
- **пароль администратора** — там же, в `appconfig` (`password_hash`, `password_salt`);
- **подготовленные статьи** и их статусы (таблица `article`).

> Сброс пароля вручную:
> ```bash
> sqlite3 data/autopost.db "UPDATE appconfig SET password_hash='', password_salt='' WHERE id=1;"
> ```

В `data/autopost.db` также лежат **сайты** (таблица `site`) и их **источники**
(таблица `source`). Секреты (ключ LLM, `SECRET_KEY`) в БД **не** хранятся — они в `.env`.

### 4.2. Сбор новостей

Новости собираются **на лету** из RSS источников сайта: если в ленте только анонс —
полная статья догружается со страницы источника, картинка берётся **из источника**
(og:image / первое изображение). Промежуточные файлы на диск не пишутся — всё идёт
сразу в обработку и сохраняется как статья в БД. Дедуп — по ссылке.

### 4.3. Готовые статьи — в `data/autopost.db`

Результат (заголовок, аннотация, текст, картинка, языки, дата публикации, статус)
хранится в таблице `article`. Статусы: `draft` → `scheduled` (есть дата) →
`published`.

## 5. Рабочий процесс

Модель: **Сайт → Источники**. Сайт — цель публикации с языками и расписанием;
источников у сайта может быть несколько (сбор идёт со всех, недоступные пропускаются).

**Настройка:**
1. Вкладка **«Сайты»** → «Создать» (название сайта).
2. На странице сайта задайте: репозиторий/ветку/путь и шаблон, **языки перевода**
   (RU/EN/CZ/DE/UA), **дни и время сбора** (напр. Пн/Пт) и **публикации**
   (напр. Ср/Вс), сколько публиковать за день, и включите автопилот.
3. Добавьте RSS-источники. Кнопка **«Просмотр»** у источника показывает живую
   ленту — видно, что поток подключён.

**Автопилот:** в дни сбора система собирает со всех источников → DeepSeek пишет
статьи → они получают статус **«Запланировано»** с датой ближайшего дня публикации.
В дни публикации запланированное публикуется. Между этим — окно правки.

**Превью по сайтам** (вкладка «Превью»): по каждому сайту секции
**Черновики / Запланировано (с датой) / Опубликовано (с датой)**. Клик по статье —
страница с **редактированием текста и даты публикации**, кнопками «Опубликовать
сейчас» и «Удалить».

**Ручной режим:** на странице сайта — «Собрать сейчас» и «Опубликовать
запланированное»; либо у источника «Просмотр» → «Обработать» конкретную новость
(создаст черновик).

Глобальные настройки модели — вкладка «Настройки»: язык, символов на новость,
«картинки строго из источника», **модель DeepSeek** (Flash/Pro), инструкция для LLM.

## 5a. Публикация на сайт

Для каждого сайта (страница сайта в админке):
1. **GitHub-репозиторий** (`owner/name`), **ветка**, **путь публикации**
   (по умолчанию `{lang}/blog/{slug}`).
2. **GitHub-токен** (PAT с правом `contents:write`) — для push в репозиторий.
3. **Шаблон статьи** — загрузите файл (Jinja2, `.j2/.html`) в разделе «Шаблон статьи»
   (или вставьте текст). По нему генератор собирает HTML каждой статьи.

При публикации генератор рендерит статью по шаблону и кладёт в репозиторий
`{path}/index.html` + `{path}/meta.json` (через GitHub API). Дальше GitHub Action
сайта заливает по FTP. Листинг/пагинацию/RSS строит PHP на сайте из этих файлов.

**Переводы.** Статья публикуется на языке генерации и на каждый выбранный язык
сайта (RU/EN/CZ/UA) — перевод через DeepSeek, общий slug, между версиями
проставляется `hreflang`. Если перевод на какой-то язык не удался, остальные
всё равно публикуются (ошибка в примечании к статье).

## 6a. HTTP API (для автоматизаций)

Включается переменной `API_KEY`. Запросы — с заголовком
`Authorization: Bearer <API_KEY>`. Базовый адрес — `http://<хост>:8778/api`.

Примеры:
```bash
curl -H "Authorization: Bearer $API_KEY" http://HOST:8778/api/status
curl -H "Authorization: Bearer $API_KEY" -H 'Content-Type: application/json' \
     -d '{"name":"mysite"}' http://HOST:8778/api/sites
curl -X POST -H "Authorization: Bearer $API_KEY" http://HOST:8778/api/sites/1/collect
curl -H "Authorization: Bearer $API_KEY" "http://HOST:8778/api/articles?site_id=1&status=scheduled"
curl -X POST -H "Authorization: Bearer $API_KEY" http://HOST:8778/api/articles/5/publish
```
Эндпоинты: `GET /api/status`, `GET/POST /api/sites`, `GET /api/sites/{id}`,
`POST /api/sites/{id}/sources`, `POST /api/sites/{id}/collect`,
`POST /api/sites/{id}/publish`, `GET /api/articles`, `GET /api/articles/{id}`,
`POST /api/articles/{id}/publish`, `POST /api/llm/test`. Управление возможно
и через админку, и через API — они работают над одними данными.

## 6. Переменные окружения (`.env`)

| Переменная | Назначение | Пример |
|------------|------------|--------|
| `LLM_PROVIDER` | Провайдер модели | `deepseek` / `hermes` / `openai` / `claude` / `local` |
| `LLM_BASE_URL` | Адрес API (обязателен для `hermes`/`local`) | `http://localhost:8642/v1` |
| `LLM_KEY` | API-ключ | `sk-...` |
| `LLM_MODEL` | Имя модели (пусто → дефолт провайдера) | `deepseek-v4-flash` / `deepseek-v4-pro` |
| `SECRET_KEY` | Подпись cookie-сессий (длинная случайная строка) | `…` |
| `DATA_DIR` | Каталог данных | `data` |
| `TZ` | Таймзона планировщика (дни/время сбора и публикации) | `Europe/Prague` |
| `API_KEY` | Ключ HTTP API (`/api/*`); пусто → API выключен | `длинная-случайная-строка` |

Проверить связь с моделью: после входа — `POST /api/llm/test`
(или кнопкой, когда добавим её в UI).

## 7. Чего пока нет

- **Реальная публикация на сайт и в соцсети** — сейчас заглушка. Для неё нужны:
  пример файла статьи на сайте (шаблон) + путь в репозитории, GitHub-токен,
  токены X/Instagram. См. `ARCHITECTURE.md` §11.
- **Планировщик** — сбор запускается кнопкой (не по расписанию).
- **Telegram-согласование** — пока только веб-превью.
